* Design Decisions
** multi-thread (loop routines) need sound coordination
Both the server and client have a routine that eventually needs to do a bunch of independent tasks without the tasks blocking each other.

For example,  the server starts by entering an =accept_loop=, given a live socket. This loop has a handler thunk which is responsible for handling clients concurrently (=handle_client_conc=). This is where we see the 3 independent tasks, which are 3 loops of their own:
 1. =read_stdin_loop=: reads user input
 2. =read_socket_loop=: reads from network
 3. =write_socket_loop=: writes to network

The loops have a ordering/logical dependency between them, that's why there's a need to handle the coordination between these threads --- which is why we need to be reliant on some *shared state*.

=Lwt= is cooperative, so no preemption worries to care about -- only need to think about coordination at the queue boundary.


*** Option A: using a =Lwt_mvar=
It's the simplest example, it's a single [[https://ocaml.github.io/odoc/lwt/lwt/Lwt_mvar/index.html][mailbox variable]]  that we can use for sync b/w the threads and in this commit it's essentially a "Message" -- whatever that came from the TCP socket.

Single-slot means =read_stdin_loop= blocks on =put= until =write_socket_loop=
calls =take=.

- Possible Issues:
  * Single-slot coupling between stdin and socket writes
  * If user types fast while network is slow, =put= blocks and typing freezes
  * Won't work for queuing multiple messages ahead of time
- Possible benefits:
  - Idiomatic Lwt
  - Creates natural backpressure: if network is slow, typing feels slow too (arguably good UX)
  - Good enough for human typing speed

*** Option B: using a =Lwt_stream=
Stream semantics, likely lazy operations as well. Non-blocking =push=. Decouples =stdin= from socket I/O completely.

- Possible Issues:
  * no explicit buffering, the stream would be unbounded
  * Internal buffering is implicit, harder to debug  -- not sure if it's bounded or unbounded queue, seems like it's lazy ops
  * can't add backpressure later to effect on UX later

- Possible benefits:
  - Push never blocks
  - Can handle bursts of messages
  - Scales to multiple producers/consumers later for extensions

*** Option C: Explicit Buffering  using a Bounded Queue
This involves us using a queue, but queues are not threadsafe so there's a need to handroll the sync for it.

- Possible Issues:
  * Needs explicit sync mechanisms --- complex code
  * might be more error-prone
  * probably no need for 1:1 chat app

- Possible benefits:
  - We will have control over the capacity of the queue
  - Can tune the backpressure manually

- unknowns:
  - are there Atomic queues or atomic vars that we can use together with Lwt? would be nice to have an atomic queue for the messages then we'd not need explicit sync via conditions or mutex

*** Option D: Using an Unbounded Queue
This is like the queue before but without a max capacity. Like a limitless buffer.

- Possible Issues:
  * unbounded memory feels like a possible source of memory leaks if rate of producing to the buffer more than rate of consuming from the buffer

- Possible benefits:
  - simple, putting into the queue will never block
  - stdin and socket writes are completely independent
*** Current Decision
For this POC, option A is good enough, but I might consider a bufferred queue because bufferred mechanics feel right in such a case of inter-thread coordination.

** Finalisation Ownership Pattern:  RAII-style "Finalisation belongs to the module that owns the resource"

Pattern discovered: "RAII intuition makes sense her Finalisation belongs to the module that owns the resource."

The intuition goes back to [[https://en.wikipedia.org/wiki/Resource_acquisition_is_initialization][RAII idiom]] which ties the lifetime of a resource directly to the lifetime of its owner: when an object goes out of scope, its destructor automatically releases any resources it owns (memory, file descriptors, locks, etc.).

*** The Anti-Pattern (Double Finalisation)

#+caption: Negative example: how double =fini= can happen
#+begin_src ocaml -n
(* Session.ml *)
let run t =
  Lwt.finalize
    (fun () -> handle_network_io t)
    (fun () -> fini t)  (* FIRST: session's own finalizer *)

(* server.ml / accept_loop *)
let%lwt () =
  Lwt.finalize
    (fun () -> Session.run session)
    (fun () -> Session.fini session)  (* SECOND: caller also finalizes *)
#+end_src

This gives a bunch of problems: double fini-callback firing, double resource cleanup, confusing errors.

*** The Fix: Clear Finalisation Boundaries
- =Session.run= owns finalisation of session state (=pending_acks=, =msg_queue=)
- =fini= callback (injected at construction of =Session=) owns finalisation of resources the session didn't create (socket, network channels [created by the session's owner])
- Caller never re-finalises what =Session.run= already finalised, doesn't expect any =fini= functions to be idempotent

The outcome is that =Session.run= is a complete, self-contained Lwt task. Caller just awaits it, doesn't finalise it.

*** Decision: Each module finalises only what it owns
- Session owns its internal state
- Fini callback (injected @ creation of session) owns the fd/socket
- Console owns its listeners, not stdin/stdout (those are process-global and borrowed @ the point when console is created)
** Resource Ownership: File Descriptors (fds) vs Lwt_io Channels

There's a decoupling between fds and the i/o channels we create to access them --- two different layers of abstraction.
This was implicit, subtle behaviour and not understanding it led to a bunch of =EBADF= errors thrown.

In my case, I wanted to be explicit about the network-resource cleanups (net-ic/oc ) @ client/server fini, but I didn't keep in mind that they're created from the same fd:
   a) for clients:- it's the client socket that is created the client network socket that is created after dns resolution and connecting to the server
   b) for server:- it's the client socket that is accepted when a client initiaties a connection that is used

The default behaviour is that the channels own the resource (fd). =Lwt_io.of_fd ~mode:(input|output) fd= creates a channel that /*owns* the fd by default/. Therefore, closing a channel will close the fd within it.
So, explicitly closing a net_ic and net_oc will lead to a double-close attempt on the fd. The second closing will lead to a EBADF error thrown.

The solution to this comes in two forms:
    A) tolerate the default (and implicit) auto-closing behaviour and avoid the double-closing problem
    B) prevent the auto-closing behaviour and decouple the ownership of the fd from the channels. This allows explicit closing of net_ic and net_oc.

I prefer the explicit action so I went with B. This means that my fini-function is the clear owner of the fd-resource-closing responsibility.

#+begin_src ocaml -n
(* NEGATIVE EXAMPLE *)
let net_ic = Lwt_io.of_fd ~mode:Lwt_io.input client_socket in   (* owns client_socket *)
let net_oc = Lwt_io.of_fd ~mode:Lwt_io.output client_socket in  (* also owns client_socket *)

(* POSITIVE EXAMPLE *)
let net_ic = Lwt_io.of_fd ~close:Lwt.return ~mode:Lwt_io.input client_socket in   (* closing channel will flush that channel only *)
let net_oc = Lwt_io.of_fd ~close:Lwt.return ~mode:Lwt_io.output client_socket in  (* closing the channel won't close the fd *)
(* By passing ~close:Lwt.return, we tell the channel: "when you close,
   flush your buffer but don't close the underlying socket". This transfers
   fd ownership to the on_fini callback. *)
#+end_src

*** The Double Close Bug
#+caption: negative example: both channels own the same fd
#+begin_src ocaml -n
(* BAD: both channels own the same fd *)
let ic = Lwt_io.of_fd ~mode:Lwt_io.input sock in
let oc = Lwt_io.of_fd ~mode:Lwt_io.output sock in
(* Closing ic closes sock; closing oc tries to close already-closed sock → EBADF *)
#+end_src

#+caption: Decoupling the ownership by making the close callback be a no-op
#+begin_src ocaml -n
(* GOOD: channels don't own the fd *)
let ic = Lwt_io.of_fd ~close:Lwt.return ~mode:Lwt_io.input sock in
let oc = Lwt_io.of_fd ~close:Lwt.return ~mode:Lwt_io.output sock in
(* Closing transfers ownership to explicit fini *)
#+end_src

*** The ~close Parameter
~close:Lwt.return tells the channel: "when you're done, flush your buffer but don't close the underlying fd."
This allows:
- Explicit, centralised fd ownership in =fini= functions
- Predictable resource cleanup without double-close bugs
- Clear mental model: the module that allocates the socket owns closing it

*** Decision
For explicit fd ownership, use =~close:Lwt.return= when creating channels from sockets.
The =on_fini= callback becomes the single point of responsibility for closing the actual socket.

**  Lwt lifecycle patterns: pick vs join, finalisation ownership
=Lwt.pick= provides coupled supervision (clean shutdown paths). =Lwt.join= provides decoupled execution (blocking behavior).

The learning is that we should choose based on:
   * do the tasks have a shared fate (both should die together) $\implies$ =Lwt.pick=,
   * do the tasks have independent fates (one dying shouldn't affect others)? $\implies$ =Lwt.join=

Finalisation must have clear ownership, remember to avoid double-finalisation by having only the owner finalise a resource $\implies$ RAII-intuition
