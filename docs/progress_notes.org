* Design Decisions

** 1. Lwt_mvar for TX coordination: natural backpressure by default

=rx_loop= and =tx_loop= are counterparts that cooperate when interfacing with the internet. That's the primary responsibility of the =Session= module. These loops need to message pass to do so, and the following options were some options:
  1. =Lwt_mvar=: single-slot mailbox
  2. =Lwt_stream=: unbounded
  3. Bounded Queue: would have to be handrolled / lib use
  4. Unbounded Queue

=Lwt_mvar= won because it's the simplest thing that could work. Single-slot means =Console.send_message= blocks on =put= until =tx_loop= calls =take=.  Meaning, it creates a natural backpressure: if the network is slow, typing feels slow. For human typing speed over TCP, this is desirable because it gives a natural UI characteristic that encodes network condition. Things should work alright for a human-timing speed application.

There's tradeoffs that are clear:
  * can't queue messages ahead of time
  * instant pasting would be limited by said backpressure

*Evolution Path:* If this was more serious, with stricter requirements, then a bus is something to look for -- streams or bounded data structures might work there.


** 2. Console lives at application scope, not session scope
<<dd_2>>

This is one of the more important architectural decisions, and it took a couple of iterations to get right.

If =Console.run= lives inside per-session handlers:
   1. It restarts per session (wasteful), losing =stdin= / =stdout= state
   2. Closing stdin on first =EOF= breaks subsequent sessions
   3. It blocks =accept_loop= from awaiting new clients

The surviving pattern:
  1. =Console= is created once at server startup, sessions are bound/unbound from it.
  2. Sessions are ephemeral within the console's lifetime.
     - This maps correctly to reality — the operator's terminal outlives any individual chat session.

A 2^{nd}-order problem comes out of this:
  1. chicken-and-egg :
     a) Console exists before any session does
     b) So there's a window where the console has no session bound.
     c) The dispatch logic handles this gracefully using  =maybe_send_msg=:  tells the operator that  "You're not in an active session" rather than crashing.

** 3. Two-phase init of Session / Console: =create= $\implies$ bind_session

A consequence of [[dd_2][the console-scope being app-level]] , =Console= and =Session= have *different lifecycles*, but they need each other to operate: =Console= needs =Session= to send messages, =Session= needs =Console= callbacks to display received messages.

The resolution is a two-phase init:
   1. =Console.create= and =Session.create= are independent
   2. =Console.bind_session= wires them together by installing callbacks

This keeps both constructors pure (no mutable state needed at creation of =Session=, =Console=) and makes the binding point explicit. The binding routine is also effectful (sets =t.session= and installs callbacks), modifying a clear state-record within =Session.t=.

The =unbind_session= counterpart exists for when a session ends but the console continues (server mode). The bind/unbind pair is the console's commit point for dispatchability -- after which they're ready to collaborate with each other.

** 4. RAII-style ownership: creator is destroyer

The intuition for RAII is second-nature but the need for it here (because of the resource management aspects) needed me to face some friction via a real bug.

Early versions had harnesses (server/client) inject =on_fini= callbacks into services (session/console). The intent was that services know /when/ teardown happens, so they should trigger cleanup. The problem with this is that services became trigger points for resources they didn't create, and this inversion of control made error propagation impossible  to be done cleanly --  services couldn't signal /why/ they exited because they were busy calling callbacks. All-to-familiar callback-hell.

The fix to this was that *each module finalises only what it owns*. Session owns its internal state (=pending_acks=, =msg_queue=). The harness owns the socket and channels it created. =Lwt.finalize= at the harness level stacks cleanups correctly.

There's some invariants to keep in mind for this:
   1. Module creates resource $\implies$ module owns cleanup
   2. Harness orchestrates $\implies$ harness stacks cleanups in its own =Lwt.finalize=
   3. Errors propagate upward $\implies$ exceptions trigger finalizers at the right level
   4. No injected callbacks $\implies$ no hidden contracts that get hard to trace

** 5. Explicit fd ownership via =~close:Lwt.return= noops @ channel-creation

This came from a funny =EBADF= bug that happens because of implicit behaviours.

=Lwt_io.of_fd ~mode:input fd= creates a channel that /owns/ the fd by default ([[https://ocsigen.org/lwt/2.6.0/api/Lwt_io][see docs]]). So if you create both =net_ic= and =net_oc= from the same socket fd, closing one channel closes the fd, and closing the other raises =EBADF=. That's mindboggling given that it's a one-liner default behaviour within the docs. The wisdom of RTFM  persists.

The fix: pass =~close:Lwt.return= when creating channels from sockets, which is just a no-op that just flushes the buffer without touching the underlying fd.  Fd ownership transfers to the harness's =fini= function, which closes the socket exactly once. This is an explicit-approach to the same ownership approach and explicit approaches are easier to ingrain than implicit behaviours.

#+caption: no-oping the callback injected for closing of the channel
#+begin_src ocaml
(* channels don't own the fd *)
let net_ic = Lwt_io.of_fd ~close:Lwt.return ~mode:Lwt_io.input sock in
let net_oc = Lwt_io.of_fd ~close:Lwt.return ~mode:Lwt_io.output sock in
(* harness fini closes sock explicitly, exactly once *)
#+end_src

This pattern is applied consistently in both =server.ml:run_single_session= and =client.ml:run_client=. The test harness uses the same pattern via =make_pipe_with_switch=.

** 6. Basic Concurrency Primitives Frames as Supervisor Policies: 3-way pick w condition variable for session supervision

=Session.run= needs to do three things concurrently:
   1. =rx_loop=: read and dispatch incoming frames
   2. =tx_loop=: take payloads from mvar, send as frames
   3. =await_shutdown_signal=: wait for explicit shutdown (e.g. operator types =/quit=)

#+caption: One for all, all for one
#+begin_src ocaml
Lwt.pick [ rx_loop t; tx_loop t; await_shutdown_signal t ]
#+end_src

=Lwt.pick= gives coupled-fate semantics: when any path completes or fails, the others are cancelled. That works well for our needs for server:  if the network dies (=rx_loop=,  =EOF=), there's no point continuing to send. If the operator quits (shutdown signal), both loops should stop.

The =Lwt_condition= approach for shutdown gives immediate cancellation without polling. A naive alternative would be a =bool ref= checked in each loop iteration, which introduces polling latency and is harder to test. The condition variable approach is idiomatic Lwt and the shutdown path is testable in isolation.

While these are relatively basic primitives to =Lwt=, it's also early in my usage of this monadic, promise-based approach to coopeerative concurrency, that's why they stood out to me and appear here.

** 7. Typed exit reasons: the single biggest testing unblock

Earlier, =Session.run= returned =unit Lwt.t=. Tests could only assert "did it run without crashing." This was blocking progress on writing meaningful tests — I kept adding spy hacks to the test harness to work around the lack of information about /why/ a session exited.

The correct response to that friction was to improve the code, not the test hacks. Now:

#+caption: Tell me, why did you leave?
#+begin_src ocaml
type exit_reason =
  | Peer_disconnected
  | Lost_conn of string
  | Broken_pipe
  | Protocol_error of Frame.error
  | Unexpected of exn

exception Session_exit of exit_reason
#+end_src

Tests can now assert =Lost_conn= vs =Protocol_error= vs clean exit with pattern matching. The same applies to Console:

#+begin_src ocaml
type error = Terminated | Channel_closed of string | Io_error of ... | Unexpected of exn
#+end_src

The harness catches these typed exits and has the opportunity to decide policy by themselves: restart session (server), keep accepting, crash gracefully. These are essentially supervision policies, and they're explicit in the code. This aspect is familiar from the Elixir world, which sometimes matches with the framing of problems here in OCaml...

** 8. Direct dispatch replaced the listener pattern

I initially built a *dynamic listener registration system* (=create_send_msg_listener=, =session_listener_factories=, =register_listeners=). It fit a /mental model of extensibility/: add behaviour by registering listeners. In practice, it added dynamic indirection without benefit.

The actual requirement of "when user types text, send it via session; when user types =/quit=, shutdown session; when no session, show a message." This is a mere static dispatch table, not a dynamic observer pattern, though.

The refactor to a direct pattern match in =run_console= was a reduction in code, a reduction in indirection, and an improvement in traceability. Each event handler is co-located and visible. Binding/unbinding of sessions became simpler.


#+begin_src ocaml -n
let run_console t =
  let break_with error =
    let msg = Format.asprintf "%a" pp_console_error error in
    let%lwt () = Lwt_io.eprintf "%s\n" msg in
    Lwt.fail (Console_exn error)
  in
  let rec loop () =
    let%lwt line = Lwt_io.read_line t.ic in
    match parse_user_input line with
    | Some User_close -> maybe_shutdown_session t >>= loop
    | Some User_exit -> Lwt.return_unit
    | Some (Send_msg bs) -> maybe_send_msg t bs >>= loop
    | Some (Slash_cmd "/help") -> show_help t >>= loop
    | None | Some (Slash_cmd _) -> loop ()
  in
  try%lwt loop () with
  | End_of_file | Lwt.Canceled -> Lwt.return_unit
  | Lwt_io.Channel_closed reason -> break_with (Channel_closed reason)
  | Unix.Unix_error (e, op, arg) -> break_with (Io_error (e, op, arg))
  | e -> break_with (Unexpected e)
#+end_src

The listener pattern would be justified if: multiple independent handlers needed to observe the same event (i.e. a fanout of event dependencies), or runtime (de)registration was required. Neither applies here. Knowing when to simplify is as important as knowing when to abstract.

** 9. Error domain separation in Frame_reader

=Frame_reader= distinguishes between two fundamentally different error categories:

#+begin_src ocaml
type error =
  | Connection_lost of string    (* network-level: EOF, ECONNRESET *)
  | Protocol_error of Frame.error (* protocol-level: bad type, oversized payload *)
#+end_src

This matters because Session can make different decisions based on which error it gets. A =Connection_lost= is an expected lifecycle event (peer left). A =Protocol_error= means *the byte stream is desynchronised and recovery is impossible*  (NOTE: for a binary TLV protocol, there's no way to find the next frame boundary after corruption, the connection has to be re-established).

This /compositional error hierarchy/ flows upward cleanly: =Frame.error= wraps into =Frame_reader.error= wraps into =Session.exit_reason=.

** 10. =Format.formatter= for error representation

Errors use =Format.formatter= (=pp_error=, =pp_exit_reason=) instead of string-based approaches. This aligns with OCaml ecosystem conventions and gives concrete benefits:

   1) Composability: nested errors format naturally via =%a= without intermediate string conversions
   2) Deferred evaluation: no allocation overhead when the error isn't printed (except if I use some hacky bridge helper functions )
   3) Testing: errors can be formatted to a buffer and asserted without hidden string construction
   4) Evolution: when error formatting gets complex, =pp= scales naturally via composition

The switch from =Failure "some string"= to =Session_exit (Protocol_error (Unknown_frame_type 0xFF))= was *the shift from stringly-typed to actually typed errors*.

* Protocol Design Notes

** The =Lwt_io= buffering bargain $\Leftarrow$ TCP saves our soul

=Frame_reader= has NO partial-read accumulation logic. This is deliberate, and it's somethign we can afford to do here :
  * =Lwt_io.read_into_exactly= handles partial TCP reads internally:  blocks until exactly =N= bytes are available, buffering across TCP segments. This is why =Frame_reader= is ~30 lines instead of ~150. =Lwt_io= is the accumulator.
  * there's a tradeoff:
    - we lose the ability to *inspect partial frames for protocol error recovery*.  For a binary TLV protocol where a malformed frame kills the session anyway (you can't resynchronise thenafter), this is the right tradeoff.

For all the other complex networking problems, we stand on the shoulder of giants and just use TCP/IP as though it's magic bytestream that just works.  Saves us having to deal with:
   1. packet drops and retransmissisons
   2. correctness checking, verification of packets...

** Int32.to_int assumes 64-bit

In =Frame.parse_header_bytes=, payload size extraction does =Bytes.get_int32_be bs payload_size_offset |> Int32.to_int=. On 64-bit Linux (the target), =int= is 63 bits so all valid =int32= values fit. On 32-bit systems, =Int32.to_int= silently truncates. This should be safe on all modern Linux deployments.

** Message IDs are 1-based

=next_msg_id= *pre-increments*: the counter initialises to =0l= and the first message gets =id = 1l=. So =id = 0l= is never used on the wire. If a peer sends =Ack { id = 0l }=, it's treated as a spurious ack (no pending message with that id). This is implicit:  no explicit reserved-id check exists.

** Max payload guard fires before allocation

=Frame_reader.read_frame= calls =parse_and_validate_header_bytes= before =Bytes.create payload_sz=. The =max_payload_size= check (1MB) fires before any allocation happens. Without this, a malicious peer sending a header with =payload_sz = 2_000_000_000= would *OOM the process* before the error was detected. This was identified during a phase where the error mode enumeration and handled correctly from the start.


* Concurrency Safety Notes

** Hashtbl safety under cooperative scheduling

=pending_acks= is a mutable =Hashtbl= accessed from two concurrent Lwt tasks: =tx_loop= (on message send: =Hashtbl.add=) and =rx_loop= (on ack receipt: =Hashtbl.find_opt=, =Hashtbl.remove=).

This is safe implicitly because of  *Lwt's is cooperative concurrency* and neither =Hashtbl.add= nor =Hashtbl.find_opt= yields to the scheduler. There are no yield points between the hashtbl access and the next yield, so no interleaving is possible.

This note feels super basic except: if this example codebase were to be migrated to =Eio= (structured concurrency with true parallelism), this would need some sort of shared-variable sync e.g. mutex or an atomic map.

** Session_meta.peer.addr is mutable, updated in background

=Session_meta= does *best-effort hostname resolution* via =Lwt.async=. The =peer.addr= field is =mutable= and is updated in-place when DNS responds (within 300ms timeout). The connection banner is displayed before DNS completes, so it shows the IP address. Any subsequent display (e.g. disconnect message) may /show the resolved hostname/. This is intentional: human-readable names are a nice-to-have, not a correctness requirement.

Under cooperative scheduling, the mutation is safe: the DNS callback runs when the current task yields, and string assignment is atomic in OCaml.

** The closed-over oc in callbacks

=make_console_callbacks= closes over =t.oc= at binding time. If =t.oc= were ever reassigned (it isn't currently), the callback would use the stale channel. This is safe because =Console.t= has no mechanism to swap output channels after creation. If that changed (e.g. terminal reconnect), callbacks would need to be rebound.


* Lowlights

There were two notable detours that cost time.

** The double-fini bug

Early versions had =Session.run= calling =Lwt.finalize= with its own fini, AND the =accept_loop= wrapping =Session.run= with another fini that called =Session.fini=. Resources were finalised twice. The second close would raise =EBADF=.

It's the classic entering of callback hell (fini functions are =unit -> unit Lwt.t=, so it's you can call them as many times as you like) so can't be protected by the OCaml's type system here.

Takeaway: feels like when fighting resource cleanup bugs, the problem is almost always unclear ownership more so than insufficient cleanup.

** The listener pattern overengineering

I built a full dynamic listener/observer system for console event dispatch (=create_send_msg_listener=, =session_listener_factories=, =register_listeners=). It took non-trivial time to wire up correctly and made the console harder to trace.

*The actual requirement was a 10-line pattern match*. The listener pattern would have been justified for plugin-style extensibility or runtime (de)registration. Neither constraints even applied here. The refactor to direct dispatch deleted code, reduced indirection, and made the console readable (and testable) again.

The lesson: classic case where problems should be solved on feeling friction, else it's just classic premature abstraction. The git history will show the cost of this for this simple chat application.

* Won't Fix (v0 Scope)

Wontfixes are choices worth making too.

** Connection timeout

No timeout on =Lwt_unix.connect= or =Lwt_unix.accept=. The =timeout= field exists in =network_config= (both server and client) but is never wired in. The correct implementation is =Lwt_unix.with_timeout= wrapping the connect/accept calls. Deprioritised really, will try to add it in.

** Logging framework

=log_level= exists in =terminal_config= but =Logs= is never initialised. All diagnostics currently go to stderr via =Lwt_io.eprintf=. The correct implementation is =Logs.set_level= at startup and =Logs.info=, =Logs.err= throughout. Omitted because the current diagnostic output is sufficient for review and the wiring is boilerplate.

The pattern of using =pp_*= pretty printers should make it easier to adapt to using =Logs= as well.

** Busy rejection for concurrent connection attempts

This part is more of an assumption on the requirements. For good UX, I would have wanted to have explicit busy-rejections when a 2nd clients attempts to connect to a server while there's an ongoing active chat session. I'm treating it as undefined behaviour for now because it's a 1:1 chat.

The evolution path for this is manageable. It CAN'T relyon the OS-backlog value because that's a best-effort value (referring to =listen server_socket 1=) -- the kernel enforces a minimum effective backlog that varies by version, so the queueing is non-deterministic.

Correct implementation for this would be to accept the connection, send a =Close= frame, close the socket. Would need a small restructuring =accept_loop= to not block on =Session.run=, *which is the multiplexing evolution path* (many active chat sessions).

For v0, the limitation is assumed to be accepted and documented here.

* Evolution Paths

Just some natural evolution paths about where teh current code could go naturally:

** Multiplexing (multiple concurrent sessions)

=Session.run= already encapsulates its lifecycle. The minimal change for multiplexing: =Lwt.async (fun () -> Session.run session)= in =accept_loop= instead of blocking on it. The accept loop keeps running, each session manages itself. The OS backlog handles queueing during async session setup.

The =Console= would need to be aware of multiple sessions (currently it holds =session : S.t option=, would need =session : S.t list= or an active-session selector). The dispatch table would need a target-session argument.

Current slash command offer a clean UI component for operators to manage sessions and so on, even though this isn't a full TUI application or anything.

** Eio migration (structured concurrency)

The architecture was designed with concurrency model swappability  in mind: cancellation semantics are explicit (=Lwt.Canceled= handlers everywhere), mutable state is documented as cooperative-only, and the supervision model (harness catches typed exits, decides policy) likely maps naturally to Eio's structured concurrency approach.

The main changes like touch the following pattern: =Lwt.pick=  $\rightarrow$ =Eio.Fiber.any=, =Lwt_mvar= $\rightarrow$ =Eio.Stream=, =Lwt_condition= $\rightarrow$ =Eio.Condition=, =Lwt_io= channels $\rightarrow$ =Eio.Flow=. The =pending_acks= Hashtbl would need a mutex or some other atomic datastructure under Eio's parallelism.

** TUI replacement (Notty/Minttea)

The current UI is line-based, like an IRC chat --  via =Lwt_io.read_line=. Console's =ic/oc= are injectable, so a Notty-based input module could replace =run_console= without changing Session, Frame, or the harnesses. The separation between user I/O (Console) and network I/O (Session) was designed with this replacement in mind.
